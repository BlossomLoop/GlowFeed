Loop Engineering：把 AI Agent 的「持续执行」纳入工程治理

让 Agent 自己盯着一个目标反复迭代、跑到做完，技术上早就不是问题。麻烦在后头：它一直在跑， 谁能保证它没乱改文件、没把活干砸、出了事能查到也能叫停？Loop Engineering 谈的就是这套配套设施—— 状态、边界、验证、预算、审计，再加一个随时把活接回人手的开关。

01它到底是什么

上游项目把 loop 定义成一种“递归目标”：你给出目的，AI 借助状态、子代理、验证和外部系统持续迭代， 直到目标达成，或者它自己判断该把问题交还给人。这跟两件听起来很像的东西得分清楚。

  • 它不是一次性的 prompt。单次问答没有跨轮次的状态，也没有“不满意就接着改”的收敛过程。
  • 它也不是所谓的“自动研发系统”。重点恰恰不在放权让 Agent 自由发挥，而在反过来给它的持续执行套一层工程约束：状态、边界、验证、预算、审计，外加人工接管。

参考实现来自开源项目 cobusgreyling/loop-engineering。 它不是一个单一运行时框架，而是一套参考体系：可复用的循环模式、按工具准备的起步模板、示例、技能与脚手架工具、以及一份生产安全边界文档。下文会逐一拆开。

02为什么需要它

一个能自己跑很多轮、还能调工具、改文件、评论 PR 的 Agent，能干多少，就能闯多少祸。没有约束的自动循环，常见的翻车有几种： 每轮依据的输入都在变，结论跟着漂；High Priority 里堆着上周早解决的旧事，越积越多；为了让 CI 过，顺手改了不该碰的文件，还没人发现。 最糟的一种，是根本没人说得清它什么时候、用什么权限、动过哪些东西。

所以这套方法的取舍很明确：先把“看得见、管得住”做扎实，再来谈放开多少自动化。也就是说，重点不是让 Agent 更能干， 而是让它每跑一轮，结果都能验、能叫停、能回滚。

下表命令仅用于识别模式，不可直接复制到生产——真实命令必须带上 state 文件、禁止项、预算、Human Gate 和运行日志要求。

04六根支柱

不管用哪个模式、哪个工具，一个能用的 loop 都得把下面六件事落到实实在在的文件和配置里。光在 Markdown 里写规则不算数，它拦不住任何东西；成熟度越高，越要把规则落到权限、凭据和 hook 这些真能生效的地方。

STATE.md

状态

循环的脊柱：每轮运行前读、运行后更新；并 prune 已解决项，防噪声堆积。

LOOP.md · denylist

边界

cadence、范围、Human Gates、禁改路径、停止条件，写清“它不能碰什么”。

loop-verifier.md

验证

maker/checker 分离的独立审查者，默认拒绝，证据不足即 ESCALATE。

loop-budget.md

预算

每日 token / 运行次数 / 子代理上限 + kill switch + 恢复条件。

loop-run-log.md

审计

append-only 运行日志：输入快照、动作、升级、token、verdict、脱敏声明。

Human Gate

人工接管

安全 / 认证 / 支付 / PII / 基建 / 依赖升级一律停下来交还人类。

关于 verifier 上游 verifier 模板的核心原则是“checker 默认拒绝，除非证据充分”。它要检查：scope（只改相关文件、无 denylist 路径）、intent（变更确实解决原目标）、tests（真的跑了，或说明为什么没跑）、no-cheating（没禁用测试、跳断言、注释掉检查）、risk（中高风险即便测试通过也建议人工 review）。 输出只能是 APPROVE / REJECT / ESCALATE_HUMAN 三选一；没有测试证据时必须 ESCALATE。LLM review 只是语义补充，测试、lint、typecheck、diff scope 才是主要证据。

05L0–L3 成熟度阶梯

别一上来就想着无人值守。成熟度得一级一级解锁，每级都有准入条件和“达不到就别往上走”的红线。loop-audit 的分数只是参考信号之一，不是唯一门槛，而且真实运行记录还得人工核一遍。

L0DRAFT

只有意图，没有执行

有 LOOP.md 初稿、目标与非目标清楚。禁止：自动运行、自动改码、自动调外部写接口。

L1REPORT-ONLY

只读分析，更新状态和报告

准入：STATE.md、triage skill、LOOP.md（cadence/范围/Human Gates）、loop-budget.md 都到位。loop-audit 建议 score ≥ 38。

升 L2 指标：连续 1–2 周稳定、误报率 < 30%、人工修正次数下降、至少一次结果被人读过并修正。

L2ASSISTED

小范围自动修复，但必过 verifier + 人工确认

准入：L1 稳定、有 loop-verifier.md 与 minimal-fix 类 action skill、改动在隔离 worktree 内、有明确测试命令与回滚策略。score ≥ 58。

允许：文档修复、单文件低风险 bugfix、补测试、lint/format、patch 级依赖（须审 lockfile diff）。禁止：自动 merge / push 主干、改 auth/payments/PII/infra/migrations、禁用测试、为过 CI 扩大范围。

L3UNATTENDED

无人值守——仅限极低风险 + 强约束 + 高可观测

准入须全满足：L2 稳定运行过一个周期、独立 worktree/runner、仅非主干分支、最小权限 token、网络 allowlist、MCP 只开必要工具、单实例锁、硬性 token/时间/子代理上限、完整日志、失败告警可达、kill switch 已验证、回滚 playbook 已演练。score ≥ 78，且默认禁止自动 merge / release。

06落地方案：在 Claude Code 上跑通第一个 Loop

最稳的起点是 Daily Triage：风险低、产出是一份报告、误报率也好观察。下面这条路径可以直接照着做，走的是上游 Claude Code starter 的主线，而不是自己再包一层 /loop-run。

第一步：目录结构

.
├── CLAUDE.md            # 项目级行为入口：规则、命令、验证要求、权限
├── LOOP.md              # 循环配置：cadence、模式、Human Gates、MCP、预算
├── STATE.md             # 状态脊柱：每轮前读、后更新
├── loop-budget.md       # token / 频率 / 子代理上限 / kill switch
├── loop-run-log.md      # append-only 运行日志
├── .claude/
│   ├── skills/
│   │   ├── loop-triage/SKILL.md
│   │   └── loop-budget/SKILL.md
│   └── agents/
│       └── loop-verifier.md     # 独立 checker，L2 及以上用
└── docs/
    └── safety.md        # denylist / auto-merge 策略 / MCP scopes / 事故响应

第二步：脚手架与体检

npx @cobusgreyling/loop-init . --pattern daily-triage --tool claude
npx @cobusgreyling/loop-cost  --pattern daily-triage --level L1 --cadence 1d
npx @cobusgreyling/loop-audit . --suggest
务必手动补 safety.md docs/safety.md 不会由 loop-init 自动生成。必须从上游 docs/safety.md 复制后项目化，或手工写一份等价的安全文档——否则首轮就该停下来。无法联网执行 npx 时，可直接从本地 starters/minimal-loop-claude/ 和 templates/ 复制。

第三步：第一周只报告（Report-only）

上游 Quickstart 明确建议：第一周不 auto-fix、不 auto-merge，只产出报告。下面是可直接粘贴的企业版首轮指令，把禁止项写死：

/loop 1d Run $loop-triage.

Before acting:
- Read CLAUDE.md, LOOP.md, STATE.md, loop-budget.md, loop-run-log.md and docs/safety.md.
- Confirm this is L1 report-only.
- Confirm no write action is allowed except updating STATE.md and appending loop-run-log.md.
- Confirm docs/safety.md exists. If missing, stop and ask the human to create it.

Scope:
- Review recent CI failures, open PRs, open issues and recent commits.
- Use only read-only access.
- Redact secrets, customer data, full environment variables and production payloads.
- Prune or archive resolved / merged / closed / stale items from STATE.md with a short reason.

Output:
- Update STATE.md: High Priority, Watch List, Noise, State Updates.
- Append one run entry to loop-run-log.md with input snapshot, branch, commit SHA, outcome, escalation count.
- Escalate anything ambiguous, or touching security / auth / payments / billing / PII / infra / dependency / production risk.

Forbidden:
- Do not edit code, create branches, commit, push, merge, close issues, change labels,
  install or upgrade packages, call production APIs, or read/write secrets.

第四步：达到 L1 完成标准

“跑起来”不等于完成。L1 收敛要满足：

  • STATE.md 每轮被读取，并更新 Last run / High Priority / Watch List / Noise / Pruned Items。
  • loop-run-log.md 有非模板的真实 run entry（run id、pattern、level、branch、commit SHA、输入快照、actions、escalations、token 估算、outcome、redaction）。
  • 无代码改动、无外部写操作；报告质量连续 1–2 周稳定；误报率可接受（建议 < 30%）。
  • 至少一次 report-only 输出被人真正读过并修正过——不能只看审计器里的 Last run 字段。

第五步：升级到 L2 的执行协议

Claude starter 默认只给 triage + verifier，不会自动装 minimal-fix。升 L2 前要：

  1. 从 templates/SKILL.md.minimal-fix 建 .claude/skills/minimal-fix/SKILL.md，写入项目测试命令、denylist 和允许修改范围。
  2. 在 LOOP.md 声明：L2 只处理 High Priority 里的低风险小项。
  3. 每次 implementer 任务给定固定输入：original target、allowed file scope、denied paths、baseline（分支 / SHA / git status / 基线测试结果）、max attempts（2–3 次）、required tests。
  4. 完成后把 diff 摘要、实际变更文件、测试命令与输出、未执行验证及原因、回滚方式交给 verifier。
  5. Verifier 只输出 APPROVE / REJECT / ESCALATE_HUMAN；无测试证据必须 ESCALATE_HUMAN，人工确认后才提交。

07权限与安全边界

这是把“规则”变成“边界”的地方。建议把高风险命令直接配成 ask 或 deny。

读取代码allowallowallow
修改代码denyask / 仅 worktreeallowlist only
git commitdenyaskallowlist only
git pushdenyask保护分支 deny
merge / releasedenydeny默认 deny
PR 评论 / 改标签denyaskallowlist only
包安装 / 升级denyask仅 patch allowlist
生产 API / 数据库写denydenydeny
secrets / env 文件denydenydeny

**/*_key*       **/*_secret*     .terraform/**     k8s/production/**
**/migrations/** auth/**          payments/**       billing/**
# 建议追加
**/prod/**      **/production/**  **/rbac/**        **/iam/**
**/policy/**    **/customer-data/**
触发以下情形必须 Human Gate 安全 / 认证 / 授权；支付 / 计费 / PII；生产基建；依赖升级（尤其 major 或高危 CVE）；改动超过 ~10 个文件；同一事项第三次失败；疑似 prompt injection 或恶意仓库内容；第三方 MCP/plugin 信任不明；日志或状态可能泄露敏感信息。

回滚与事故响应

任何 L2/L3 变更前记录：当前分支、commit SHA、git status、变更目标、允许范围、基线测试结果；变更后保留：diff 摘要、测试结果、verifier verdict、回滚方式（依赖升级再加 package/lockfile diff 与 downgrade 路径）。一旦合入坏代码：立即拉 kill switch → 暂停调度 → revert → 记事故到 run-log → 收紧 scope/denylist/verifier/预算后再恢复。

08周度复盘指标

L1 升 L2 前必须有至少一轮周度复盘记录。每周盯这几条：

指标阈值 / 目标
False positives（误报率）< 30%
False negatives（漏报）趋势下降
Auto-fixes proposedL1 必须为 0
RollbacksL1 必须为 0
Human escalations有明确 owner
Token spend estimate不超过 budget
Mean time to human awareness趋势下降

09套用到本项目（GlowFeed）

把方法论落到手边的系统才算真落地。GlowFeed 本身是个有调度器、多信源抓取的服务， 天然适合一个 L1 report-only 的 daily-triage，目标是巡检抓取健康度而非改任何代码：

  • 目的：每天读 data/ 下的运行记录与 /api/runs，盯三件事——是否有任务当天抓取为 0、各信源失败/超时率、入库增量是否异常掉零。
  • 状态：一份 STATE.md 记 High Priority（如“微博源连续 2 天 0 抓取”）、Watch List、Noise；每轮 prune 已恢复的源。
  • 边界：只读 data/ 与本地 API，禁止改 app/、改 config.json、动 admin_token、调外部源的写操作。
  • 预算：每日 1 次、token 上限给足余量、L1 子代理数为 0。
  • 人工接管：发现疑似信源被封、或 token/密钥相关异常，直接升级，不自行处置。

它跟现有的“调度器启动补跑错过的计划任务”正好互补：调度器保证任务在跑，这个 loop 负责让人知道任务跑得好不好、并留下记录。等连续两周稳定、误报率也降到能接受了，再考虑要不要升到 L2，做点像“给失效信源自动打个标记”这种极小的修复，而且照样得过 verifier。

10落地验收清单

  • 已基于 loop-init --tool claude 或等价结构准备 .claude/skills 与 loop-verifier.md。
  • CLAUDE.md 含/引用 loop 规则；LOOP.md 定义 cadence、scope、Human Gates、worktree、MCP、budget。
  • STATE.md 有明确 schema；loop-budget.md 有每日 cap、子代理上限、kill switch。
  • loop-run-log.md 有输入快照与审计字段；docs/safety.md 已手工创建/项目化。
  • L1 明确禁止代码修改与外部写；L2 明确 minimal-fix、worktree、verifier、人工确认；L3 明确硬权限、网络 allowlist、单实例锁、告警与回滚。
  • 敏感信息治理明确（脱敏 / gitignore / 受控系统）；外部凭据最小权限；Runtime Profile 已定义身份、凭据、权限、日志、通知与暂停方式。
  • loop-audit . --suggest 主要问题已处理或记录例外，且 activity 已人工核验；周度 metrics 面板已建立。

说到底，Loop Engineering 的意义不在给 Agent 更多权限，而在把它的持续执行管起来。状态、权限、验证、预算、审计、回滚这几样都闭环了，loop 才算从“提示词自动化”变成一条能放心交付的工程流程。想动手的话，从 daily-triage 的 L1 report-only 开始就够了。
